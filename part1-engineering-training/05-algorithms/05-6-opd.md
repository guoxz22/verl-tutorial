OPD（On-Policy Distillation）是 v0.8.0 中最值得单独学习的新能力之一。它不是 PPO/GRPO 的一个普通参数，也不是离线 SFT；它把“学生模型自己 rollout”与“教师模型给 token 级监督”结合起来。

一句话理解：

```text
OPD = 学生按当前策略生成轨迹 + 教师在这些学生轨迹上给 logprob/top-k 分布 + 学生更新
```

## 为什么需要 OPD

普通知识蒸馏通常让学生学习教师生成的数据。问题是：训练时学生看到的是教师轨迹，推理时学生却会进入自己生成的状态。一旦学生和教师的推理路径不同，就会出现 exposure bias。

RLVR/GRPO 解决了“状态来自学生自己”的问题，因为 rollout 来自当前学生策略；但奖励通常是稀疏的，比如整题对/错。OPD 把两者结合起来：

| 方法 | 状态来自谁 | 监督信号 | 常见问题 |
| --- | --- | --- | --- |
| SFT / 离线 KD | 教师或数据集 | token 级 | 训练/推理状态不一致 |
| GRPO / RLVR | 学生 | 结果级 reward | 奖励稀疏，reward 设计成本高 |
| OPD | 学生 | 教师 token logprob / top-k 分布 | 需要额外教师推理资源 |

## 训练流程

一个 OPD step 可以拆成四步：

```text
1. 从数据集中取 prompt
2. 学生 actor rollout，生成 response
3. 教师模型读取 prompt + 学生 response 前缀，返回每个 token 位置的监督信号
4. 学生用 distillation loss 更新
```

这也是它名字里 on-policy 的含义：训练用的状态来自学生当前策略，而不是预先采样好的教师轨迹。

## 与 PPO / GRPO 的关系

v0.8.0 的 OPD 仍然走 `python -m verl.trainer.main_ppo` 入口，并复用 actor / rollout / trainer 配置；但它额外打开 `distillation.*` 命名空间。

| 维度 | PPO | GRPO | OPD |
| --- | --- | --- | --- |
| 训练入口 | `main_ppo` | `main_ppo` | `main_ppo` |
| 常用 advantage | `gae` | `grpo` | 示例脚本常用 `grpo` |
| 是否需要 Critic | 通常需要 | 通常不需要 | 通常不需要 |
| 主要监督 | reward + advantage | group reward | teacher logprob / top-k 分布 |
| 额外资源 | Reward Model 可选 | reward function | teacher resource pool |

注意：`algorithm.adv_estimator=grpo` 在 OPD 示例中仍然可能出现，但它不等于“OPD 就是 GRPO”。GRPO 负责 advantage/reward 侧的估计方式；OPD 的核心开关是 `distillation.enabled=True` 和 `distillation.distillation_loss.*`。

## 最小单教师脚本

下面的脚本保留官方 FSDP 示例的关键结构，但把变量压到最少，便于第一次理解。

```bash
#!/usr/bin/env bash
set -xeuo pipefail

STUDENT_MODEL=${STUDENT_MODEL:-Qwen/Qwen3-8B}
TEACHER_MODEL=${TEACHER_MODEL:-Qwen/Qwen3-32B}

train_files="['$HOME/data/gsm8k/train.parquet','$HOME/data/math/train.parquet']"
val_files="['$HOME/data/gsm8k/test.parquet','$HOME/data/math/test.parquet']"

python3 -m verl.trainer.main_ppo \
  algorithm.adv_estimator=grpo \
  algorithm.use_kl_in_reward=False \
  data.train_files="$train_files" \
  data.val_files="$val_files" \
  data.train_batch_size=128 \
  data.max_prompt_length=1024 \
  data.max_response_length=2048 \
  data.filter_overlong_prompts=True \
  data.truncation=error \
  data.shuffle=False \
  actor_rollout_ref.model.path="$STUDENT_MODEL" \
  actor_rollout_ref.model.use_remove_padding=True \
  actor_rollout_ref.model.enable_gradient_checkpointing=True \
  actor_rollout_ref.actor.optim.lr=1e-6 \
  actor_rollout_ref.actor.ppo_mini_batch_size=128 \
  actor_rollout_ref.actor.use_dynamic_bsz=True \
  actor_rollout_ref.actor.ppo_max_token_len_per_gpu=24576 \
  actor_rollout_ref.actor.fsdp_config.param_offload=True \
  actor_rollout_ref.actor.fsdp_config.optimizer_offload=True \
  actor_rollout_ref.rollout.name=vllm \
  actor_rollout_ref.rollout.tensor_model_parallel_size=2 \
  actor_rollout_ref.rollout.gpu_memory_utilization=0.4 \
  actor_rollout_ref.rollout.n=1 \
  actor_rollout_ref.rollout.max_model_len=3073 \
  distillation.enabled=True \
  distillation.n_gpus_per_node=4 \
  distillation.nnodes=1 \
  distillation.teacher_models.teacher_model.model_path="$TEACHER_MODEL" \
  distillation.teacher_models.teacher_model.inference.name=vllm \
  distillation.teacher_models.teacher_model.inference.tensor_model_parallel_size=2 \
  distillation.teacher_models.teacher_model.inference.gpu_memory_utilization=0.4 \
  distillation.teacher_models.teacher_model.inference.max_model_len=3073 \
  distillation.distillation_loss.loss_mode=k1 \
  distillation.distillation_loss.topk=64 \
  distillation.distillation_loss.use_task_rewards=False \
  distillation.distillation_loss.use_policy_gradient=True \
  distillation.distillation_loss.loss_max_clamp=10.0 \
  distillation.distillation_loss.log_prob_min_clamp=-10.0 \
  trainer.n_gpus_per_node=8 \
  trainer.nnodes=1 \
  trainer.val_before_train=False \
  trainer.save_freq=200 \
  trainer.test_freq=5 \
  trainer.total_epochs=15 \
  trainer.logger='["console","wandb"]' \
  trainer.project_name=verl_distill_gsm8k_math \
  trainer.experiment_name=qwen3_8b_from_qwen3_32b_fsdp
```

这里最容易忽略的是资源池：`trainer.n_gpus_per_node` 是训练主资源，`distillation.n_gpus_per_node * distillation.nnodes` 是教师资源池。v0.8.0 会校验教师池大小，要求它等于所有 teacher replicas 的 GPU 总需求。

## 关键配置

| 配置 | 含义 | 常用写法 |
| --- | --- | --- |
| `distillation.enabled` | 是否开启 OPD | `True` |
| `distillation.n_gpus_per_node` | 教师资源池每节点 GPU 数 | 单机常写教师占用 GPU 数 |
| `distillation.nnodes` | 教师资源池节点数 | 开启时至少为 `1` |
| `distillation.teacher_models.teacher_model.model_path` | 单教师模型路径 | `Qwen/Qwen3-32B` |
| `distillation.teacher_models.<name>.key` | 多教师路由 key | 与样本的 `data_source` 等字段对应 |
| `distillation.teacher_models.<name>.num_replicas` | 教师副本数 | 多教师时显式设置 |
| `distillation.teacher_key` | 多教师路由字段 | 默认 `data_source` |
| `distillation.distillation_loss.loss_mode` | 蒸馏损失模式 | `k1` / `k3` / `forward_kl_topk` |
| `distillation.distillation_loss.topk` | top-k 教师分布大小 | `32` 或 `64` |
| `distillation.distillation_loss.use_policy_gradient` | 是否把蒸馏信号作为 PG reward | `k1` 常配 `True` |
| `distillation.distillation_loss.use_task_rewards` | 是否混合任务 reward | 纯蒸馏常配 `False` |

## loss_mode 怎么选

v0.8.0 的配置和代码里有两类思路：

| 方式 | 推荐配置 | 直觉 |
| --- | --- | --- |
| PG OPD | `loss_mode=k1` + `use_policy_gradient=True` | 把教师/学生 logprob 差转成 token 级 reward，再走 policy gradient |
| Supervised OPD | `loss_mode=k3` 或 `forward_kl_topk` + `use_policy_gradient=False` | 直接把教师分布作为监督信号反传 |

不要把 `forward_kl_topk` 和 `use_policy_gradient=True` 当成默认搭配。源码中的 `DistillationLossConfig` 会提示：`forward_kl_topk` 更适合作为 supervised distillation loss，因为 PG 路径主要使用采样 token 的 logprob，难以充分利用 top-k 分布信号。

## Megatron / VeOmni 怎么改

FSDP 脚本不需要设置 `model_engine`。如果换训练后端，主要改这些地方：

| 后端 | 关键差异 |
| --- | --- |
| Megatron | 加 `model_engine=megatron`，并配置 `actor_rollout_ref.actor.megatron.tensor_model_parallel_size` / `pipeline_model_parallel_size` |
| VeOmni | 加 `model_engine=veomni`，并配置 `actor_rollout_ref.actor.veomni.*` |
| VeOmni fused top-k | 当使用 `forward_kl_topk` 这类 top-k 分布损失时，可关注 `actor_rollout_ref.model.use_fused_kernels=True` |

无论训练后端如何，教师推理配置仍在 `distillation.teacher_models.<name>.inference.*` 下，形状类似 rollout 配置。

## 多教师 OPD

多教师适合把数学、代码、多模态等不同领域教师合成到同一个学生模型中。基本写法是新增多个 teacher entry：

```bash
+distillation.teacher_models.math.key=openai/gsm8k \
+distillation.teacher_models.math.model_path=Qwen/Qwen3-32B \
+distillation.teacher_models.math.num_replicas=1 \
+distillation.teacher_models.math.inference.name=vllm \
+distillation.teacher_models.math.inference.tensor_model_parallel_size=2 \
+distillation.teacher_models.code.key=code \
+distillation.teacher_models.code.model_path=<code-teacher> \
+distillation.teacher_models.code.num_replicas=1 \
distillation.teacher_key=data_source
```

注意：默认的 `teacher_models.teacher_model` 是单教师便利入口。做多教师时不要把它和其他 teacher entry 混用；建议统一改成 `math`、`code`、`vl` 这类显式名字。

## 常见问题

### 问题 1：开启 OPD 后资源池报错

检查：

- `distillation.nnodes` 是否大于 0。
- 教师池总 GPU 数是否等于所有教师副本需要的 GPU 总数。
- `num_replicas * tensor_model_parallel_size * data_parallel_size * pipeline_model_parallel_size` 是否算对。

### 问题 2：教师模型 OOM

处理顺序：

1. 降低 `distillation.teacher_models.teacher_model.inference.gpu_memory_utilization`。
2. 降低 `distillation.distillation_loss.topk`。
3. 减小 prompt/response 长度，并同步调整 teacher inference 的 `max_model_len`。
4. 增加 teacher TP 或减少同机训练 GPU 占用。

### 问题 3：不知道该用 OPD 还是 GRPO

如果有可靠可验证 reward，GRPO 更简单；如果有强教师但 reward 稀疏或难设计，OPD 更自然。也可以用 `use_task_rewards=True` 把任务 reward 和蒸馏信号混合，但第一次跑通建议先用纯蒸馏配置降低变量数量。
